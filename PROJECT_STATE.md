# PROJECT_STATE.md — Healthcare Appointment & Follow-up Manager

Tracks cross-module state. Read before starting any module.

## Stack (locked)
- Frontend: Next.js 14 App Router + TS + Tailwind → Vercel (Hobby)
- Backend: Node + Express + TS → Render (free Web Service, spins down after 15min, cold-start 30-60s — accepted)
- DB: PostgreSQL via Prisma → Neon (free scale-to-zero)
- Queue: Redis + BullMQ → Upstash (free, 256MB, 500K cmd/mo, ioredis TCP/TLS URL — NOT REST)
- Auth: self-implemented JWT with `role` claim
- Email: Resend (3000/mo, 100/day); fallback Brevo via Nodemailer/SMTP (NOT SendGrid)
- Calendar: Google Calendar API v3 + OAuth 2.0 via `googleapis`
- LLM: NVIDIA NIM (OpenAI-compatible @ https://integrate.api.nvidia.com/v1), ~40 req/min, called by deployed app only

## Rules (non-negotiable)
1. Three roles only: patient / doctor / admin. Server-side role + ownership checks.
2. No duplicate booking on (doctor_id, date, time) — DB UNIQUE constraint inside tx.
3. Doctor leave on a date with existing bookings → detect BEFORE commit, notify every patient.
4. Symptom form is the final step before Confirm. Booking write MUST NOT depend on/rollback for LLM/email/calendar — all three best-effort async.
5. LLM failure → 1 retry with backoff, then neutral fallback stored + shown. Never block/break.
6. Email (booking confirm/remind/cancel) to both patient + doctor, via queue w/ retry + visible failure state. Never sync in request handler.
7. Medication reminders computed by background job from prescription frequency, not on-demand.
8. Google Calendar events: create on booking, update on reschedule, delete on cancel. Failures never block/rollback booking.
9. Don't invent endpoints/fields/flows. State assumptions explicitly.
10. TS end-to-end, no `any` without a one-line justification comment.
11. Conflicting instruction → follow the rule above, flag the conflict in output.

## M6 Notes
- **OAuth 2.0 flow**: `GET /calendar/connect` (auth) constructs Google URL w/ `access_type:'offline'`, scope `https://www.googleapis.com/auth/calendar.events`, `prompt:'consent'`, and a single-use CSRF `state` (Redis `oauth_state:{state}`, TTL 600s). After consent, browser redirects to `GET /calendar/callback` (PUBLIC) w/ `code`+`state`. Callback validates the state against Redis (consuming it immediately so replay fails), exchanges the code via `OAuth2Client.getToken(code)`, encrypts with AES-256-GCM, upserts `OauthToken` keyed on `userId` (`Schema:391`). HTML stub returned (with optional `FRONTEND_OAUTH_RETURN_URL` redirect after 2s).
- **Tokens at rest**: AES-256-GCM with `OAUTH_TOKEN_ENC_KEY` (32 raw bytes, base64-encoded). Storage format: `base64(iv(12) || ciphertext || tag(16))` — fits the existing `access_token`/`refresh_token` text columns. Decryption failure (GCM auth tag mismatch / wrong key) surfaces a "re-connect required" message.
- **Refresh logic** (`oauthService.getValidAccessToken`): decrypts `access_token`, compares `expiryDate` against `now + 60s` skew buffer. If expired: acquire `oauth_refresh_lock:{userId}` Redis SET-NX (TTL 30s) to serialise simultaneous refresh calls for one user. Second caller short-polls for the lock holder's DB update at 50ms × 6 (300ms) and returns the recently-refreshed row without another Google call. Lock holder calls `client.refreshAccessToken()`, re-encrypts + writes back via `prisma.oauthToken.update`, returns the new access token. **If Google returns `invalid_grant`** (user revoked our app in their Google account settings): catch → delete the local `OauthToken` row → return `null` → caller (worker) silently skips that party. **Other errors (5xx / network)**: rethrown so BullMQ can retry the whole job.
- **Disconnect**: `POST /calendar/disconnect` calls `OAuth2Client.revokeToken(refreshToken)` (best-effort, errors swallowed in `try/catch(console.warn)`), then `prisma.oauthToken.delete({where:{userId}})` unconditionally. Does NOT clear `CalendarEvent.*EventId` values — pre-existing calendar events orphaned on the user's calendar can still be cleaned by future cancel jobs if the user reconnects.
- **Calendar API payloads (create)**: `calendar.events.insert({calendarId:'primary', requestBody, sendUpdates:'none'})`. `requestBody.summary`: `"Appointment with Dr. {doctorFullName}"` for the patient-side event, `"Appointment: {patientFullName}"` for the doctor-side event. `description`: 8-char booking ref + cancellation hint + automation footer (no UUIDs, no PII). `start/end`: `dateTime` ISO + `timeZone:'UTC'`; `end = start + DoctorProfile.slotDurationMinutes` minutes (default 30). `attendees`: single email = the connected party's own email, `responseStatus:'accepted'` (deliberately does NOT add the other party as an attendee — avoids leaking the other party's email across calendars without consent and avoids invite emails). `reminders.overrides`: popup 60min + email 1440min (24h, aligned with M5 booking-reminder cadence). `source.url`: `FRONTEND_OAUTH_RETURN_URL` if set, otherwise `null` — never undefined (strict mode in `tsconfig`). No symptom form, chief complaint, severity, allergies, prescriptions, or doctor notes are ever sent.
- **Calendar API payloads (delete)**: `calendar.events.delete({calendarId:'primary', eventId, sendUpdates:'none'})`. Normalises gaxios errors: 404 / 410 → idempotent success (event already gone); other 4xx → throws `GooglePermanentError` (worker marks `*CalendarConnected=false` and clears the stored event id so it cannot retry forever); 5xx / network / timeout → rethrown for BullMQ retry.
- **Worker (`calendarSyncWorker`)**: Same Upstash Redis connection as M4/M5 via `getRedisConnection()` (`redis.ts:53`). `Queue('calendar-sync')` w/ `attempts: 3, backoff: {type:'exponential', delay:30_000}, removeOnComplete:200, removeOnFail:false`. Worker `concurrency: 2` (env-overridable via `CALENDAR_WORKER_CONCURRENCY`). Two ops distinguished by `job.data.op`: `create` → `handleCreate`; `delete` → `handleDelete`. Both re-read the full booking + CalendarEvent row from DB at execute time (enqueue payload carries only `bookingId` — never trust a pre-commit snapshot).
- **Trigger wiring (Rule 8)**: All three M3 trigger functions retain `: void` and are called without `await`. M6 added `calendarSyncQueue.add(...).catch(log)` calls alongside (not replacing) the existing M4 LLM `preVisitQueue.add(...).catch(log)` and M5 notification `enqueue*Notifications(ctx).catch(log)` calls. Each calendar dispatch is in its own `try { ... } catch (console.error)` block inside the trigger function — so a calendar dispatch failure cannot touch the email or LLM dispatch and cannot propagate to the booking response. **(Reschedule is two independent jobs**, not one, so the delete and create failures are isolated from each other.)
- **Failure state**: `CalendarEvent.syncStatus` enum values `PENDING`/`SYNCED`/`RETRYING`/`FAILED` (all reserved in M1 schema; M6 uses them, did not need a migration). Visible-failure pattern: worker `on('failed')` handler checks `job.attemptsMade >= job.opts.attempts`, and on exhaustion calls `prisma.calendarEvent.updateMany({where:{bookingId}, data:{syncStatus:'FAILED', lastSyncError:'BullMQ retries exhausted: ...'}})`. No new table; reuses M1's reserved `CalendarEvent` table as the calendar analog of M5's `Notification` table.
- **Idempotency**: `CalendarEvent.bookingId @unique` makes row-upsert idempotent. On any re-entry of the create job (e.g. after a partial failure where the patient event id was stored but doctor event id was not), the upsert overwrites `syncStatus` to `RETRYING` and the next pass inserts the missing id. Delete handler clears `*EventId` to `null` after success, making subsequent retries idempotent.
- **Visible failure surfacing**: there is no admin UI for `syncStatus='FAILED'` rows in this module (Rule 9 — not invented). Row inspection by SQL query suffices for now; a future module can build an admin portal panel without schema change.
- **Queue isolation**: M5 email queue name `email-notification` and M6 calendar queue name `calendar-sync` share the same Upstash instance and BullMQ connection but are isolated by BullMQ's per-queue key prefixing (`bull:<queueName>:*` in Redis). No key collision possible; both workers can run concurrently in the same Node process.
- **Rule 11 divergence recorded (A9(M6))**: the literal wording of Rule 8 says "updated on reschedule". M6 implements **delete-old + create-new** two-job pattern instead of `events.patch` because (a) M3's reschedule creates a new `Booking` row and (b) `CalendarEvent.bookingId @unique` (`schema.prisma:369`) requires one row per booking. Patching would require either a schema migration (forbidden — M1–M5 code frozen) or migrating the `CalendarEvent` row from `oldBookingId` to `newBookingId` in the same transaction as the reschedule (forbidden — Rule 4 forbids async/calendar work in the booking tx). User-visible effect is identical (event shows at new time on user's calendar); ICS subscriptions may render "cancelled + new invite" rather than "moved event".
